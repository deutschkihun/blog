## 1. MSA (Micro Service Architecture)

과거에는 모든 서비스 어플리케이션 및 api (결제,정산,조회 등등) 같은 것들을 하나의 단일 시스템으로 관리 했음 (모놀리스)

쉽게 말하면 여러개의 기능을 가능 하나의 서비스를 마이크로 단위로 기능개발을 하여 최종적으로 배포할 떄 하나로 합쳐서 작동하게 끔 하는 아키텍처 구조이다.

하나의 서비스를 마이크로 단위로 나누어 개발하게 되면 추후에 다른 서비스에 재활용하는 것도 가능하며 이러한 아키텍쳐는 하나의 마이크로 서비스의 오류 및 에러가 서비스 전체에 영향을 미치는 경우를 방지 할 수 있다.

기존의 모놀리스 방식을 마이크로 방식으로 쪼갬 (a,b,c,d 서비스)
a,b,c,d 서비스 api 를 이용하여 통신, 복잡도는 당근 올라감, 비용 올라감 (api 호출이 많아지니 서버에 자원사용량 올라가기 떄문에 비용도 올라감)

그럼 뭐가 좋음 ??

- 마이크로 단위로 나누어서 서비스를 관리하니 전체서비스중 일부에 장애가 생겨 작동하지 않는 경우 서비스를 마이코로 단위로 모듈화 하였기 때문에 부분장애가 모든 서비스로 전파 되지 않음, 장애가 있는 부분만 문제가 있고 나머지는 정상작동함 (가장큰 장점)

- 예를들어 서비스의 회원이 점점 늘어나고 서비스 사용량과 인프라도 증가 하는 중. 기존의 api 기능중 조회 기능이 전체 기능중 90프로 이상 차지함. 그래서 마이크로 단위로 설계 헀기 때문에 조회 관련 서비스 인프라부분만 더 확충하면 서비스 괸리가 가능함 (한마디로 정리하면 효율적인 인프라 관리)

- 조회라는 기능만 예를 들어 1주일마다 조회 기능이 업데이트 되어 배포 된다고 했을때 마이크로 단위로 관리해주면 조회 서비스만 새롭게 배포해주고 나머지는 정상 작동하게 두면 되며 서비스 전체를 빌드하지 않아도 되고 의존성 문제로 해결 할 수 있음 (해당 하는 부분만 배포해줄수 있음) -> devops (빈번한 배포가 생겨남 = 애자일) / 토스 발표자료 참고
